# Experiment Report: Clock Arithmetic Baseline & Proof of Training (Run 0–1)

> [!IMPORTANT]
> All results use a **43-probe subset** (pre-subtraction, before SB-01–SB-05 were added to `validate.py`). Current `TEST_CASES` contains 48 probes. Numbers are frozen to the 43-case eval runs stored in `results/`.

## Objective
Establish a baseline for HH:MM clock arithmetic in small language models (SmolLM-135M) and validate the SFT approach using structured `[THINK]` scratchpad blocks.

## Run 0: Zero-Shot Baseline
- **Model**: `HuggingFaceTB/SmolLM-135M` (135M parameters)
- **Method**: Direct zero-shot evaluation across 43 diagnostic probes.
- **System Prompt**: "You are a precise time calculator. Given a starting time and a duration, calculate the resulting time. Respond with ONLY the final time in HH:MM 24-hour format. Do not explain your reasoning."
- **Results**:
    - Exact Match: 0/43 (0.0%, 95% CI [0.0–8.2%])
    - Normalized Match: 1/43 (2.3%, 95% CI [0.4–12.1%])
    - Structural Match: 0/43 (model has never seen scratchpad format)
- **Failure Breakdown**: BASE_10_ERROR: 0, TOKEN_COLLAPSE: 2, FORMAT_HALLUCINATION: 3, OTHER_ERROR: 37
- **Finding**: The dominant failure mode is OTHER_ERROR — the model echoes inputs, generates irrelevant text, or produces non-time outputs. BASE_10_ERROR (base-10/base-60 confusion) was not observed at this model scale.

## Phase 0: Data Hardening
- **Corpus**: 56,938 synthetic records generated by `generator.js` using Luxon-powered deterministic arithmetic.
- **Diversity**: 12 domains (Domestic, Tech, Logistics, etc.) balanced at ~7% each.
- **Features**: 
    - 11.5% Format Jitter (Spaced digits) for tokenization hardening.
    - 6.3% Subtraction logic (Backwards temporal reasoning).
    - Shadow Pair adjacency for contrastive math learning.

## Run 1: Proof of Training (PoT)
- **Model**: SmolLM-135M (QLoRA, 1,000 iterations, 5K sample).
- **System Prompt (CoT)**: "You are a precise time calculator. Given a starting time and a duration, calculate the resulting time. Show your reasoning in a [THINK] block, then give the final time in HH:MM 24-hour format."
- **Results**:

    | Metric | Score | 95% CI | Notes |
    |--------|-------|--------|-------|
    | Normalized Match | 0/43 (0.0%) | [0.0–8.2%] | Standard normaliser cannot parse scratchpad outputs |
    | Oracle Match | 7/43 (16.3%) | [8.1–30.0%] | Parser heuristic, not model capability — see [audit](../README.md#oracle-audit) |
    | Structural Match (loose) | 40/43 (93.0%) | [81.4–97.6%] | ≥2 scratchpad elements (including delimiters) |
    | Structural Match (tight) | 28/43 (65.1%) | [50.2–77.6%] | Addition + mod/carry/hour (arithmetic substance required) |

- **Structural Match Definitions**:
    - **Loose**: `is_structural_match(raw)` — fires on ≥2 of: `\d+\+\d+=\d+`, `mod60=\d+`, `carry=\d+`, `H:\d+\+\d+=\d+`, `[THINK]`, `[RESULT]`, `[ANSWER]`. Can inflate with delimiter-only matches.
    - **Tight**: `is_structural_match_tight(raw)` — requires at least one addition pattern (`D+D=D`) AND at least one `mod60`/`carry`/`H:D+D=D` pattern. Delimiter-only outputs do not qualify.

- **Oracle Match Caveat**: The oracle parser (`normalize_time_oracle`) is a regex heuristic that extracts computed times from `mod60=` and `H:X+Y=Z` patterns. It can credit the model when patterns coincidentally align with gold. All 7 hits have been manually audited (see [oracle audit](../README.md#oracle-audit)) — each contains unambiguous, correct arithmetic. But the metric measures "how often does the parser happen to extract the right answer from the trace," not "how often does the model reason correctly."

- **Key Observations**:
    1. The model learned scratchpad format at 93% loose / 65.1% tight. The tight number (requires arithmetic substance) is the more defensible claim.
    2. Standard normalised match is 0% because the model emits arithmetic traces without producing a clean `HH:MM` final answer.
    3. Oracle parsing recovers 7 correct answers (16.3%). The gap between oracle (16.3%) and standard (0%) is the "answer extraction" problem.
    4. 15 outputs (93% − 65.1% = 28%) have scratchpad structure but lack arithmetic substance — they contain delimiters or simple addition without carry logic.

- **Direct-vs-CoT Confound**: The direct and CoT modes use different system prompts. Direct mode instructs "do not explain" while CoT instructs "show reasoning in a [THINK] block." This means direct-vs-CoT comparisons are not purely ablating the presence of a scratchpad.

## Conclusions
1. SFT teaches format (65.1% tight structural match) but not end-to-end accuracy (0% normalised match).
2. Oracle parsing shows partial arithmetic competence (16.3%) masked by a formatting bottleneck — though this is a parser capability metric, not a model capability metric.
3. BASE_10_ERROR is not the primary failure mode at this model scale. OTHER_ERROR dominates.
4. These results motivate external computation (Route-to-Luxon) rather than continued SFT attempts at this model scale.
